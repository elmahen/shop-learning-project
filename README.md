## Project Setup (Target State)

### Scenario A – Development (DB in Docker, Backend + Frontend local)

This is the standard setup during development.

**1. Start the database**
```bash
docker compose -f docker-compose.db.yml up -d
```

> **How the database user is created**  
> PostgreSQL's Docker image automatically creates the database user, password and database on first startup — driven by the environment variables in `docker-compose.db.yml`:
> ```yaml
> POSTGRES_DB:       shopdb      # database name
> POSTGRES_USER:     shop_user   # application user (not a superuser)
> POSTGRES_PASSWORD: shop_pass   # password
> ```
> No manual setup needed. The user is the owner of the database and has full privileges on it.  
> This is why we do **not** use the default `postgres` superuser for the application.

**2. Load schema and test data**
```bash
./db/schema.sh
./db/testdata.sh
```

**3. Open an interactive psql session**
```bash
./db/psql.sh
```

**4. Reset database (drop → schema → testdata)**
```bash
./db/reset.sh
```

**5. Start backend** *(available from Iteration 2)*
```bash
cd backend
./mvnw spring-boot:run
```

**6. Start frontend** *(available from Iteration 5)*
```bash
cd frontend
npm install
npm run dev
```

## Git Workflow

### Branch Convention

One feature branch per iteration:

```
iteration/01-pure-sql
iteration/02-data-access
iteration/03-service-layer
iteration/04-rest-api
iteration/05-react-ui
iteration/06-dockerization
```

### Workflow per Iteration

```bash
# 1. Create branch
git checkout -b iteration/01-pure-sql

# 2. Work, commit regularly
git add .
git commit -m "feat: add customer table with constraints"

# 3. Push branch
git push -u origin iteration/01-pure-sql

# 4. Open Pull Request on GitHub → Code Review
# 5. Address feedback on the same branch
# 6. After approval → merge into main
```

### Commit Message Convention

```
feat:      new functionality
fix:       bug fix
docs:      documentation
test:      tests
refactor:  refactoring without functional change
```

Examples:
```
feat: add customer table with unique email constraint
fix: correct foreign key on order_items
docs: add UML class diagram iteration 1
test: add repository integration tests for customer
refactor: extract payment calculation into PaymentService
```
